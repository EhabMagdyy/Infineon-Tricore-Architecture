# 1. Introduction

AURIX microcontrollers contain one of the most advanced interrupt architectures used in automotive systems.

Unlike traditional microcontrollers where peripherals communicate directly with an interrupt controller, TriCore introduces an intermediate layer called the:

```
Service Request Node (SRN)
```

This provides:

* Flexible interrupt routing
* Multi-core support
* Fine-grained priority control
* DMA integration
* Improved scalability

The interrupt architecture is designed specifically for:

* Engine control
* Electric vehicles
* ADAS
* Autonomous driving
* Vehicle networking

---

# 2. Interrupts vs Traps

Many engineers confuse interrupts and traps.

Although both transfer control to a handler, they originate from different sources.

---

## Interrupt

Generated by hardware peripherals.

Examples:

```
CAN RX

ADC Conversion Complete

Timer Overflow

Ethernet Packet Received
```

Interrupts are typically asynchronous.

---

## Trap

Generated internally by the CPU.

Examples:

```
Divide By Zero

Illegal Instruction

Context Error

Memory Protection Violation
```

Traps are generally synchronous to instruction execution.

---

# 3. Why TriCore Interrupts Are Different

Traditional Cortex-M Architecture:

```
Peripheral
    |
    v
  NVIC
    |
    v
   ISR
```

TriCore Architecture:

```
Peripheral Event
        |
        v
      SRN
        |
        v
Interrupt Router
        |
        v
     CPUx
        |
        v
      ISR
```

The Service Request Node sits between the peripheral and CPU.

This enables significantly more flexibility.

# 5. Service Request Nodes (SRN)

The Service Request Node is one of the most important concepts in AURIX.

Think of it as:

```
Intelligent Interrupt Object
```

An SRN contains information about:

* Priority
* Enable State
* Request Pending Status
* Target CPU
* Routing Information

Instead of a peripheral interrupting the CPU directly, it raises a request through its SRN.

---

## Example: CAN Module

A CAN controller may generate:

```
CAN RX

CAN TX

CAN Error

CAN Bus-Off

CAN FIFO Full
```

Internally:

```
CAN Module

   |
   +---- RX SRN

   |
   +---- TX SRN

   |
   +---- ERROR SRN

   |
   +---- FIFO SRN
```

Each event is independently configurable.

---

## Example: ADC Module

ADC may generate:

```
Conversion Complete

Queue Empty

Limit Check

Result Ready
```

Internally:

```
ADC

   |
   +---- SRN0

   |
   +---- SRN1

   |
   +---- SRN2

   |
   +---- SRN3
```

Every event can have its own priority and routing.

---

# 6. Service Request Control Registers (SRC)

Each SRN is configured through a Service Request Control Register.

Common examples:

```
SRC_CAN0RX0

SRC_ASCLIN0TX

SRC_GTM_TIM0
```

These registers control interrupt behavior.

---

## Conceptual SRC Layout

```
+-------------------+
| SRPN              |
+-------------------+
| TOS               |
+-------------------+
| SRE               |
+-------------------+
| SRR               |
+-------------------+
```

Where:

```
SRPN = Priority

TOS = Target CPU

SRE = Enable

SRR = Request Pending
```

# 7. Interrupt Routing

One of TriCore's most powerful features.

An interrupt can be routed to a specific CPU.

Example:

```
CAN RX
   |
   v
  SRN

   |
   +----> CPU0
```

or

```
   +----> CPU1
```

or

```
   +----> CPU2
```

This is configured through TOS.

---

# 8. Type of Service (TOS)

TOS determines:

```
Who handles the request?
```

Possible targets:

```
CPU0

CPU1

CPU2

CPU3

CPU4

CPU5

DMA
```

Example:

```
TOS = CPU2
```

Result:

```
CAN RX
   |
   v
  CPU2
```

No involvement from other CPUs.

---

# 9. Interrupt Priorities (SRPN)

SRPN:

```
Service Request Priority Number
```

Priority Range:

```
0 - 255
```

Example:

```
Brake ISR        = 250

Motor Control    = 220

CAN RX           = 120

UART RX          = 20
```

Higher value means:

```
Higher Priority
```

---

## Priority Example

```
Priority 250

      |
      v

Priority 220

      |
      v

Priority 120

      |
      v

Priority 20
```

The highest pending request executes first.

# 11. Interrupt Vector Table

The interrupt vector table maps priorities to handlers.

Conceptually:

```
Interrupt Vector Table

0x80000000 -> Timer ISR

0x80000020 -> CAN ISR

0x80000040 -> ADC ISR
```

When an interrupt occurs:

```
Interrupt
     |
     v
Vector Lookup
     |
     v
   ISR
```

---

# 12. Interrupt Service Routines (ISR)

Typical ISR declaration:

```c
IFX_INTERRUPT(CanRxISR, 0, 120)
{
}
```

Meaning:

```
ISR Name = CanRxISR

CPU      = 0

Priority = 120
```

The startup code automatically places the ISR into the vector table.

---

# 13. Nested Interrupts

Higher-priority interrupts can interrupt lower-priority interrupts.

Example:

```
Task

  |
  +--> CAN ISR

            |
            +--> BRAKE ISR
```

Because:

```
BRAKE Priority > CAN Priority
```

Execution order:

```
Task

  -> CAN ISR

       -> BRAKE ISR

       <- Return

  <- Return
```

Back to Task

This allows critical events to be serviced immediately.

# 14. Software Interrupts

Interrupts can be generated by software.

Example:

```
CPU0

   |
   v

Trigger ISR
```

Applications:

* Testing
* Synchronization
* Inter-core communication

# 15. DMA as Interrupt Target

Not every event must wake a CPU.

Traditional Flow:

```
ADC
  |
  v
 CPU
  |
  v
 RAM
```

DMA Flow:

```
ADC
  |
  v
 DMA
  |
  v
 RAM
```

Configure:

```
TOS = DMA
```

Benefits:

* Zero CPU involvement
* Lower latency
* Reduced CPU load

---

# 16. Multi-Core Interrupt Routing

Example System:

```
CPU0 = Vehicle Control

CPU1 = CAN Gateway

CPU2 = Ethernet

CPU3 = Diagnostics
```

Routing:

```
CAN RX      -> CPU1

Ethernet RX -> CPU2

Diagnostics -> CPU3

Control Loop -> CPU0
```

This dramatically improves scalability.

# 17. Trap Architecture

Interrupts originate from peripherals.

Traps originate from the CPU.

Conceptually:

```
Interrupt
     |
     +--> Peripheral Generated

Trap
     |
     +--> CPU Generated
```

---

# 18. Trap Classes

TriCore defines eight trap classes.

---

## Class 0

Memory Management Traps

Examples:

```
Address Translation Errors
```

---

## Class 1

Internal Protection Traps

Examples:

```
Privilege Violations

Protected Resource Access
```

---

## Class 2

Instruction Error Traps

Examples:

```
Illegal Opcode

Invalid Instruction
```

---

## Class 3

Context Management Traps

Examples:

```
CSA Exhaustion

Context Chain Errors
```

---

## Class 4

System Bus Traps

Examples:

```
Bus Timeout

Bus Access Error
```

---

## Class 5

Assertion Traps

Examples:

```
Hardware Consistency Violations
```

---

## Class 6

System Call Traps

Equivalent to:

```
Supervisor Call
```

Used by operating systems.

Applications:

```
AUTOSAR OS Services
```

---

## Class 7

Non-Maskable Interrupt (NMI)

Highest priority fault handler.

Examples:

```
Critical Safety Faults

Fatal Hardware Errors
```

---

# 19. Trap Vector Table

Separate from interrupt vectors.

Conceptually:

```
Trap Table

Class 0 -> Handler

Class 1 -> Handler

Class 2 -> Handler

Class 3 -> Handler

Class 4 -> Handler

Class 5 -> Handler

Class 6 -> Handler

Class 7 -> Handler
```

Flow:

```
Fault

  |
  v

Trap Class

  |
  v

Trap Handler
```

# 20. Trap Handling Examples

---

## Example 1: Divide By Zero

Code:

```c
int x = 10;
int y = 0;

int z = x / y;
```

Flow:

```
Division

  |
  v

CPU Detects Error

  |
  v

Trap Generated

  |
  v

Trap Handler
```

---

## Example 2: Illegal Instruction

CPU fetches:

```
Invalid Opcode
```

Result:

```
Instruction Error Trap
```

---

## Example 3: Memory Protection Violation

Application accesses:

```
Protected Memory
```

Result:

```
Internal Protection Trap
```

---

## Example 4: CSA Exhaustion

No free CSA available.

```
FCX = NULL
```

Another context save required.

Result:

```
Class 3 Trap
```

Typical action:

```
Log Error

Enter Safe State

Reset System
```

# 21. Interrupt vs Trap Comparison

| Feature      | Interrupt       | Trap           |
| ------------ | --------------- | -------------- |
| Source       | Peripheral      | CPU            |
| Example      | CAN RX          | Divide By Zero |
| Asynchronous | Yes             | Usually No     |
| Routing      | SRN             | Trap Class     |
| Vector Table | Interrupt Table | Trap Table     |
| Priority     | SRPN            | Trap Class     |
| Purpose      | Event Handling  | Fault Handling |
